т# Варианты решения задачи поиска статей

## Постановка

Дано: 793 статьи справки (title + body в HTML), 500 размеченных запросов (calibration.f), 500 тестовых запросов (test.f).
Метрика: MAP@10.
Ограничения: без внешних LLM/API, локальные модели разрешены, воспроизводимость.

---

## 1. Обработка текста (применимо ко всем подходам)

### 1.1 Очистка HTML
Из body нужно извлечь чистый текст.

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| `beautifulsoup4.get_text()` | Стандарт, обрабатывает любую разметку | Медленнее lxml |
| `lxml.html.fromstring().text_content()` | Быстрее bs4 | Может оставить мусор |
| `markdownify` (html→md) | Убирает разметку, сохраняет структуру | Лишняя зависимость |
| Просто strip тегов через re | Нет зависимостей | Ломается на сложном HTML (таблицы, спойлеры) |

### 1.2 Префиксная схема для dense retrieval

Модели семейства E5 (intfloat/multilingual-e5-*) требуют префиксов:
- query: `"query: {query_text}"`
- passage: `"passage: {title} {body_text}"`

ruBERT / LaBSE / cointegrated/rubert — без префиксов.

## 2. Алгоритмы ранжирования

### 2.1 TF-IDF (word-level)

**Как работает:** TfidfVectorizer на слова (стемминг через nltk SnowballStemmer или pymorphy2), косинусная близость.

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| Стеминг (SnowballStemmer) | Быстрый, уменьшает словарь | Грубый, может испортить редкие слова |
| Лемматизация (pymorphy2) | Точнее стемминга | Медленнее (~1 мс/слово), для 793 статей — ОК |
| Без морфологии | Максимально просто | Пропускает «оплатить» ≠ «оплачу» |

**Вердикт:** дешёвый baseline для пристрелки, MAP@10 будет заметно ниже dense retrieval.

### 2.2 TF-IDF (character n-grams)

**Как работает:** `TfidfVectorizer(analyzer='char', ngram_range=(2, 5))`.

| Плюсы | Минусы |
|-------|--------|
| Устойчив к опечаткам («оплатитьь» → «оплатить») | Размерность признаков большая (тысячи n-gram) |
| Не нужна морфология (схватывает падежи как общие n-gram) | Не понимает семантику (синонимы, перефразирования) |
| Не требует external моделей | Слаб на длинных текстах, хорош только на коротких (заголовки) |

**Вердикт:** хорош как **первый baseline для заголовков** (они короткие, 29 символов в среднем). Не требует external зависимостей — только sklearn.

### 2.3 BM25 (Okapi BM25)

**Как работает:** `rank_bm25` или имплементация вручную. Стеминг + BM25 scoring.

| Плюсы | Минусы |
|-------|--------|
| Часто лучше TF-IDF на коротких запросах | Не семантический |
| Учитывает длину документа (короткие title не штрафуются) | По умолчанию не обрабатывает русскую морфологию |
| | Для заголовков (~5 слов) TF-IDF и BM25 дают почти одно и то же |

**Вердикт:** для title body разница с TF-IDF минимальна. Стоит попробовать, если переходим на body.

### 2.4 Dense Retrieval (Sentence Transformers)

**Как работает:** эмбеддер → косинусная близость между эмбеддингом запроса и эмбеддингами статей.

| Модель | Размер | Качество | Примечание |
|--------|--------|----------|------------|
| `intfloat/multilingual-e5-small` | 256-dim | Хорошее | Multilingual, префиксы query:/passage:, ~250MB |
| `intfloat/multilingual-e5-base` | 768-dim | Лучше | ~1GB |
| `cointegrated/rubert-tiny2` | 768-dim | Среднее | Русскоязычная, быстрая (~80MB) |
| `DeepPavlov/rubert-base-cased-sentence` | 768-dim | Высокое | Под русский, ~1.5GB |
| `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` | 384-dim | Хорошее | Проверенный мультиязычный, ~500MB |
| `intfloat/multilingual-e5-large-instruct` | 1024-dim | Лучшее | Тяжёлая (~2.5GB) |

**Вердикт:** основной кандидат. MAP@10 с e5-small обычно даёт +15-20% к BM25.

### 2.5 Гибрид (BM25 + Dense)

**Как работает:** взвешенная сумма нормализованных скоров: `score = α · BM25_score + (1-α) · dense_score`.

| Плюсы | Минусы |
|-------|--------|
| Объединяет точность BM25 (буквальные совпадения) и семантику dense | Дополнительный параметр α, нужно подбирать |
| Устойчивее к опечаткам (BM25 + char n-grams покрывает, dense уточняет) | Два индекса → больше памяти |
| Почти всегда даёт прирост vs каждый по отдельности | |

**Вердикт:** рекомендуется как основной «рабочий» вариант. α обычно 0.2–0.4 для dense-heavy.

### 2.6 Кросс-энкодер реранжирование

**Как работает:** BM25/dense → retrieve top-K (50-100) → кросс-энкодер попарно (query, candidate) → реранжирование.

| Модель | Плюсы | Минусы |
|--------|-------|--------|
| `cross-encoder/ms-marco-MiniLM-L-6-v2` | Высокое quality | Не multilingual |
| `DiTy/cross-encoder-rubert-v2` | Русскоязычный | Тяжёлый |
| `cross-encoder/stsb-rubert` | Под русский | Для STS, не для поиска |

**Вердикт:** может дать +3-5% MAP, но inference на 500 запросов × 50 кандидатов = 25K пар. Если кросс-энкодер тяжёлый, будет ~1-3 мин на CPU.

## 3. Дополнительные техники улучшения

### 3.1 Query rewriting (локальной LM)

**Идея:** маленькая генеративная модель переформулирует запрос пользователя, выделяя чистую интенцию.

```
«Здравствуйте! Я не предлагала скидку на эту брошь при данной цене. Это предложение выслано автоматически? Как убрать скидку?»
→ «как отменить скидку»
```

| Модель | Размер | Примечание |
|--------|--------|------------|
| `ai-forever/rugpt3small` | 125M | Быстрая, ~200MB |
| `ai-forever/rugpt3medium` | 355M | ~1.5GB |
| `google/mt5-small` | 300M | Multilingual, ~1GB |
| `Qwen/Qwen2.5-0.5B-Instruct` | 500M | Хорошо следует инструкциям |

**Плюсы:**
- Схлопывает «как оплатить кроссовки» и «как оплатитьь попугая» в одну интенцию
- Убирает шум (приветствия, лишние детали)

**Минусы:**
- Модель может ошибиться с интенцией → эмбеддинг уедет в сторону
- Inference ~0.5-2 сек/запрос на CPU
- Нужна генеративная модель ~0.5-2GB

**Воспроизводимость:** да, `temperature=0, do_sample=False, seed=42`.

### 3.2 Doc2Query (генерация synthetic queries)

**Идея:** для каждой статьи генерируем 3-5 вероятных вопросов пользователей и добавляем их в индекс (как дополнительные документы с тем же article_id) или используем для обучения эмбеддера.

```
Статья «Отправить заказ»:
→ «Как отправить товар через Авито?»
→ «Что делать после того, как товар заказали?»
→ «Не могу отправить заказ, что делать?»
```

| Плюсы | Минусы |
|-------|--------|
| Генерация один раз (не на инференсе) | Качество генерации маленькой модели может быть низким |
| Наполняет индекс разговорными формулировками | Шум от неудачных генераций |
| Решает проблему vocabulary mismatch | ~2000-4000 дополнительных текстов на 793 статьи |

**Воспроизводимость:** да, при фиксированном seed.

### 3.3 Rule-based очистка запроса

**Идея:** без модели, просто регулярками убрать приветствия и лишнее.

```python
import re
def clean_query(q):
    q = re.sub(r'(?i)(здравствуйте|добрый день|добрый вечер|подскажите|пожалуйста|спасибо)', '', q)
    q = re.sub(r'\s+', ' ', q).strip()
    return q
```

| Плюсы | Минусы |
|-------|--------|
| Максимально просто, без external зависимостей | Не убирает контекст («я не предлагала скидку на эту брошь») |
| Быстро, детерминированно | Не схлопывает синонимичные интенции |

**Вердикт:** рекомендую делать всегда — это бесплатно и почти всегда немного улучшает MAP.

### 3.4 Расширение запроса (query expansion)

**Идея:** добавить к запросу синонимы или связанные слова из тезауруса.

| Источник | Плюсы | Минусы |
|----------|-------|--------|
| WordNet/PWN для русского | Мало данных | Устаревший |
| Автоматический (через word2vec/fasttext) | Покрывает сленг | Нужна дополнительная модель |
| Из названий разделов/категорий Авито | Точно релевантно | Нужно их сначала извлечь |

**Вердикт:** сомнительно, лучше Doc2Query даст то же самое качественнее.

### 3.5 Использование body (полный текст статьи)

**Идея:** не только title, но и body статьи.

| Плюсы | Минусы |
|-------|--------|
| Больше контента для матчинга | HTML шум, нужно чистить |
| Статьи с одинаковым title (4× «Частые вопросы») различаются по body | Длинные тексты → больше шума при dense encoding |
| Лучше recall для специфичных запросов | Дольше encode (~10-50× длиннее) |

**Стратегии:**
- Только title — быстро, базовая линия
- Title + body — больше контента
- Title + body (first N chars) — баланс скорости и качества
- Разные эмбеддинги для title и body и weighted sum

**Вердикт:** body стоит добавить после того, как baseline на title готов. MAP@10 почти всегда растёт, но не всегда значительно.

### 3.6 Ансамбль индексов

**Идея:** 
- Один индекс: `title_embedding` (e5-small)
- Второй индекс: `cleaned_body_tfidf` (BM25 или char n-grams)
- Третий индекс: `synthetic_queries_embedding` (e5-small)
- Финальный скор: взвешенная сумма трёх

**Вердикт:** максимальное качество, но больше кода. Стоит, если предыдущие шаги упёрлись в потолок.

## 4. Рекомендуемая стратегия

### Фаза 1 — быстрый baseline
1. Очистка запросов (rule-based)
2. Dense retrieval `multilingual-e5-small` на **title**
3. Оценка MAP@10 на calibration.f

### Фаза 2 — улучшение
4. Добавить body (title + cleaned body)
5. Сравнить MAP
6. Попробовать гибрид (dense + BM25 на char n-grams)

### Фаза 3 — продвинутые техники (если не хватает качества)
7. Doc2Query (генерация synthetic queries, добавление в индекс)
8. Query rewriting на тестовых запросах
9. Кросс-энкодер реранжирование

Каждый шаг валидируется на calibration.f. Фиксируем конфигурацию, которая дала лучший MAP@10, и применяем к test.f.
